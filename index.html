💡 How It All Fits Together

A model training run starts with Slurm / Kubernetes scheduling jobs across hundreds of DGX H100 nodes. Inside each node, 8× H100s communicate at 900 GB/s via NVLink 4. Across nodes, InfiniBand NDR 400G with SHARP in-network reduction handles all-reduce collective ops. The training framework (Megatron-LM + DeepSpeed) sits atop PyTorch 2, using NCCL 2 for collectives and cuDNN / Flash Attention 3 for kernel execution. After training, TensorRT-LLM compiles the model for inference, and NVIDIA Triton (or vLLM) serves it behind a load balancer at millions of RPS.

🧭Developer Journey — How the Layers Integrate

This is the practical path a developer follows: start by validating the GPU node, write model code in PyTorch, scale it with distributed training, then compile and deploy the resulting model behind a production endpoint.

1. Hardware 2. Network 3. System Software 4. ML Framework 5. Training 6. Inference 7. Orchestration
1

Hardware — confirm the node is usable

Before writing any training or serving code, a developer checks what GPUs are available and whether CUDA can see them.

  • Touchpoint: nvidia-smi, CUDA runtime, torch.cuda
  • Produces: a confirmed GPU target for every layer above it
shell + python 
$ nvidia-smi --query-gpu=name,memory.total,driver_version --format=csv
name, memory.total [MiB], driver_version
NVIDIA H100 80GB HBM3, 81480 MiB, 550.54.15

import torch

assert torch.cuda.is_available()
print(torch.cuda.get_device_name(0))
print(torch.cuda.get_device_properties(0).total_memory // 1024**3, "GB")
Integration: every higher-level tool here ultimately targets the GPU discovered in this step.
2

Network — wire GPUs together with NCCL

Once multiple nodes are involved, developers usually interact with the network indirectly through NCCL and torch.distributed.

  • Touchpoint: InfiniBand interface selection, rank metadata, process group init
  • Produces: the communication fabric training and inference runtimes rely on
shell + python 
export NCCL_SOCKET_IFNAME=ib0
export NCCL_DEBUG=INFO
export MASTER_ADDR=10.0.0.10
export MASTER_PORT=29500

import os
import torch.distributed as dist

dist.init_process_group(
    backend="nccl",
    init_method=f"tcp://{os.environ['MASTER_ADDR']}:{os.environ['MASTER_PORT']}",
)
print("world size:", dist.get_world_size())
Integration: NCCL chooses NVLink inside a node and InfiniBand/RoCE across nodes so the model code does not have to.
3

System software — call CUDA, cuBLAS, and Flash Attention

Most developers do not call cuBLAS or cuDNN directly; they write tensor code that lowers into those libraries.

  • Touchpoint: CUDA tensors, compiled kernels, fused attention ops
  • Produces: fast GPU kernels reused by the framework and training stack
python 
import torch
import torch.nn.functional as F

q = torch.randn(1, 16, 1024, 128, device="cuda", dtype=torch.bfloat16)
k = torch.randn(1, 16, 1024, 128, device="cuda", dtype=torch.bfloat16)
v = torch.randn(1, 16, 1024, 128, device="cuda", dtype=torch.bfloat16)

with torch.backends.cuda.sdp_kernel(enable_flash=True, enable_math=False, enable_mem_efficient=False):
    out = F.scaled_dot_product_attention(q, k, v)

gemm = torch.compile(lambda a, b: a @ b)
result = gemm(torch.randn(4096, 4096, device="cuda"), torch.randn(4096, 4096, device="cuda"))
Integration: these tensor ops are where PyTorch hands work down to CUDA, cuBLAS, cuDNN, and Flash Attention.
4

ML framework — write the model in PyTorch

The ML framework is the layer most application developers actively author against: modules, losses, optimizers, and dataloaders.

  • Touchpoint: nn.Module, autograd, optimizers, token tensors
  • Produces: a trainable model graph that the distributed runtime can scale out
python 
import torch
from torch import nn

class TinyBlock(nn.Module):
    def __init__(self, hidden=4096):
        super().__init__()
        self.attn = nn.MultiheadAttention(hidden, num_heads=32, batch_first=True)
        self.mlp = nn.Sequential(
            nn.Linear(hidden, hidden * 4),
            nn.GELU(),
            nn.Linear(hidden * 4, hidden),
        )

    def forward(self, x):
        x = x + self.attn(x, x, x, need_weights=False)[0]
        return x + self.mlp(x)

model = TinyBlock().cuda().bfloat16()
tokens = torch.randn(8, 1024, 4096, device="cuda", dtype=torch.bfloat16)
loss = model(tokens).float().mean()
Integration: framework code expresses intent; CUDA libraries from Step 3 make it fast; distributed runtimes in Step 5 make it scale.
5

Training — scale the toy model to many GPUs

At this point a developer stops thinking about one process and starts thinking about tensor, pipeline, and data parallelism.

  • Touchpoint: torchrun, Megatron-LM, DeepSpeed, FSDP config
  • Produces: checkpoints that can later be exported to an inference engine
shell 
torchrun \
  --nproc_per_node=8 \
  --nnodes=2 \
  --node_rank=$NODE_RANK \
  pretrain_gpt.py \
  --tensor-model-parallel-size 8 \
  --pipeline-model-parallel-size 2 \
  --micro-batch-size 2 \
  --global-batch-size 128 \
  --bf16 \
  --use-flash-attn \
  --distributed-backend nccl \
  --save /checkpoints/tiny-gpt
Integration: the training launcher consumes the model from Step 4 and the NCCL fabric from Step 2, then emits checkpoints for inference.
6

Inference — turn checkpoints into an optimized engine

After training, the artifact changes shape: a research checkpoint becomes a serving artifact optimized for latency, throughput, and memory efficiency.

  • Touchpoint: TensorRT-LLM build/export commands or an engine like vLLM
  • Produces: a runnable model package for a serving framework
shell 
python convert_checkpoint.py \
  --input_dir /checkpoints/tiny-gpt \
  --output_dir /artifacts/tiny-gpt-hf

trtllm-build \
  --checkpoint_dir /artifacts/tiny-gpt-hf \
  --output_dir /engines/tiny-gpt-fp8 \
  --gemm_plugin fp8 \
  --max_batch_size 32 \
  --max_input_len 4096 \
  --max_output_len 1024
Integration: this layer consumes training checkpoints and packages them so Triton, NIM, or another serving tier can load them.
7

Orchestration — deploy the serving stack on a cluster

Finally, developers hand the optimized engine to infrastructure tooling so the model becomes a stable service with health checks and autoscaling.

  • Touchpoint: Triton model repository, Kubernetes, DCGM exporter, service endpoints
  • Produces: a production API that callers can hit over HTTP/gRPC
yaml + shell 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: triton-llm
spec:
  template:
    spec:
      containers:
        - name: triton
          image: nvcr.io/nvidia/tritonserver:24.10-py3
          args: ["tritonserver", "--model-repository=/models"]
          resources:
            limits:
              nvidia.com/gpu: 1

curl http://triton-llm.default.svc.cluster.local:8000/v2/models/tiny-gpt/generate \
  -d '{"text_input":"Explain NVLink in one sentence","max_tokens":64}'
Integration: orchestration binds together the inference artifact from Step 6, the GPU nodes from Step 1, and observability such as DCGM and Prometheus.

End-to-end mental model

A developer writes PyTorch code, which dispatches into CUDA / cuBLAS / cuDNN on top of the detected GPU hardware. When that model scales beyond one node, NCCL uses NVLink and InfiniBand to move tensors. A distributed trainer such as Megatron-LM saves checkpoints, TensorRT-LLM converts them into an optimized engine, and Triton + Kubernetes turns that engine into a production endpoint.

🔬Distributed Training Stack

Training a 70B+ parameter LLM requires 3D parallelism across hundreds to thousands of GPUs. These frameworks manage that complexity.

3D Parallelism Explained

Data Parallelism (DP): Replicate model; each replica processes a different mini-batch. Gradients all-reduced across replicas.

Tensor Parallelism (TP): Split individual layers (e.g., QKV attention, MLP) across GPUs. Requires all-reduce each layer.

Pipeline Parallelism (PP): Partition sequential model layers across GPUs. Microbatches overlap compute and communication.

Example — GPT-3 175B: TP=8 (within DGX node via NVLink), PP=16 (across nodes via IB), DP=2048 (data parallelism)

⚡Inference & API Serving Stack

Serving a 70B model at millions of requests per day requires continuous batching, KV-cache optimisation, quantization, and multi-node tensor parallelism.

🏎️

TensorRT-LLM (NVIDIA)

NVIDIA's LLM inference SDK. In-flight batching, paged KV cache, FP8/INT4 quantization, speculative decoding, tensor and pipeline parallelism. 4–8× vs PyTorch baseline.

NVIDIA
🎛️

NVIDIA Triton Inference Server

Production-grade model serving framework. Supports TensorRT-LLM, PyTorch, ONNX backends. Dynamic batching, multi-model serving, health probes, gRPC + HTTP API.

NVIDIA
⚡

vLLM

UC Berkeley project. Introduced PagedAttention for efficient KV-cache management — up to 24× more throughput than HuggingFace Transformers. OpenAI-compatible API, most popular OSS inference engine.

Open Source
🤗

TGI (HuggingFace)

Text Generation Inference. Continuous batching, tensor parallelism, AWQ/GPTQ/bitsandbytes quantization. Powers HuggingFace Inference Endpoints.

Open Source
🌐

SGLang

Stanford / LMSYS project. RadixAttention for efficient prefix KV cache sharing, continuous batching, multi-modal. Fastest for long-context and batch inference scenarios.

Open Source

NVIDIA NIM

Pre-packaged, cloud-ready inference microservices. Docker containers with TensorRT-LLM + Triton pre-configured for specific models (LLaMA-3, Mistral, etc.). Deploy in minutes on any NVIDIA-certified cloud.

🏭Data Center Scale Architecture

Scaling to thousands of GPUs requires a carefully designed fat-tree network topology with InfiniBand or RoCE at every layer.

Interconnect Bandwidth Hierarchy

⚡

NVLink 4 / 5 (Intra-node)

H100: 900 GB/s bidirectional between GPUs within a DGX node via NVSwitch 3. GB200 NVL72: 1.8 TB/s per GPU via NVLink 5 and NVSwitch 4, 130 TB/s total across 72 GPUs.

900 GB/s – 1.8 TB/s
🌐

InfiniBand NDR 400G

400 Gb/s per port. DGX H100 has 8× HDR/NDR ports = 3.2 Tb/s per node. Quantum-2 switches connect up to 800 ports non-blocking.

400 Gb/s/port
🔷

RoCE v2 (Ethernet alternative)

RDMA over Converged Ethernet. 400G or 800G options. More cost-effective than IB; used by hyperscalers (Meta, Google). Requires tuning for congestion control.

400–800 Gb/s
📡

SHARP (In-network Reduction)

Scalable Hierarchical Aggregation and Reduction Protocol. Performs AllReduce inside IB switches, avoiding sending data to host. 2–5× improvement in collective communication at scale.

IB Feature
