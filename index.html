💡 How It All Fits Together

A model training run starts with Slurm / Kubernetes scheduling jobs across hundreds of DGX H100 nodes. Inside each node, 8× H100s communicate at 900 GB/s via NVLink 4. Across nodes, InfiniBand NDR 400G with SHARP in-network reduction handles all-reduce collective ops. The training framework (Megatron-LM + DeepSpeed) sits atop PyTorch 2, using NCCL 2 for collectives and cuDNN / Flash Attention 3 for kernel execution. After training, TensorRT-LLM compiles the model for inference, and NVIDIA Triton (or vLLM) serves it behind a load balancer at millions of RPS.

🧭Developer Journey — How the Layers Integrate

This is the practical path a developer follows: start by validating the GPU node, write model code in PyTorch, scale it with distributed training, then compile and deploy the resulting model behind a production endpoint.

1. Hardware 2. Network 3. System Software 4. ML Framework 5. Training 6. Inference 7. Orchestration
1

Hardware — confirm the node is usable

Before writing any training or serving code, a developer checks what GPUs are available and whether CUDA can see them.

  • Touchpoint: nvidia-smi, CUDA runtime, torch.cuda
  • Produces: a confirmed GPU target for every layer above it
shell + python 
$ nvidia-smi --query-gpu=name,memory.total,driver_version --format=csv
name, memory.total [MiB], driver_version
NVIDIA H100 80GB HBM3, 81480 MiB, 550.54.15

import torch

assert torch.cuda.is_available()
print(torch.cuda.get_device_name(0))
print(torch.cuda.get_device_properties(0).total_memory // 1024**3, "GB")
Integration: every higher-level tool here ultimately targets the GPU discovered in this step.
2

Network — wire GPUs together with NCCL

Once multiple nodes are involved, developers usually interact with the network indirectly through NCCL and torch.distributed.

  • Touchpoint: InfiniBand interface selection, rank metadata, process group init
  • Produces: the communication fabric training and inference runtimes rely on
shell + python 
export NCCL_SOCKET_IFNAME=ib0
export NCCL_DEBUG=INFO
export MASTER_ADDR=10.0.0.10
export MASTER_PORT=29500

import os
import torch.distributed as dist

dist.init_process_group(
    backend="nccl",
    init_method=f"tcp://{os.environ['MASTER_ADDR']}:{os.environ['MASTER_PORT']}",
)
print("world size:", dist.get_world_size())
Integration: NCCL chooses NVLink inside a node and InfiniBand/RoCE across nodes so the model code does not have to.
3

System software — call CUDA, cuBLAS, and Flash Attention

Most developers do not call cuBLAS or cuDNN directly; they write tensor code that lowers into those libraries.

  • Touchpoint: CUDA tensors, compiled kernels, fused attention ops
  • Produces: fast GPU kernels reused by the framework and training stack
python 
import torch
import torch.nn.functional as F

q = torch.randn(1, 16, 1024, 128, device="cuda", dtype=torch.bfloat16)
k = torch.randn(1, 16, 1024, 128, device="cuda", dtype=torch.bfloat16)
v = torch.randn(1, 16, 1024, 128, device="cuda", dtype=torch.bfloat16)

with torch.backends.cuda.sdp_kernel(enable_flash=True, enable_math=False, enable_mem_efficient=False):
    out = F.scaled_dot_product_attention(q, k, v)

gemm = torch.compile(lambda a, b: a @ b)
result = gemm(torch.randn(4096, 4096, device="cuda"), torch.randn(4096, 4096, device="cuda"))
Integration: these tensor ops are where PyTorch hands work down to CUDA, cuBLAS, cuDNN, and Flash Attention.
4

ML framework — write the model in PyTorch

The ML framework is the layer most application developers actively author against: modules, losses, optimizers, and dataloaders.

  • Touchpoint: nn.Module, autograd, optimizers, token tensors
  • Produces: a trainable model graph that the distributed runtime can scale out
python 
import torch
from torch import nn

class TinyBlock(nn.Module):
    def __init__(self, hidden=4096):
        super().__init__()
        self.attn = nn.MultiheadAttention(hidden, num_heads=32, batch_first=True)
        self.mlp = nn.Sequential(
            nn.Linear(hidden, hidden * 4),
            nn.GELU(),
            nn.Linear(hidden * 4, hidden),
        )

    def forward(self, x):
        x = x + self.attn(x, x, x, need_weights=False)[0]
        return x + self.mlp(x)

model = TinyBlock().cuda().bfloat16()
tokens = torch.randn(8, 1024, 4096, device="cuda", dtype=torch.bfloat16)
loss = model(tokens).float().mean()
Integration: framework code expresses intent; CUDA libraries from Step 3 make it fast; distributed runtimes in Step 5 make it scale.
5

Training — scale the toy model to many GPUs

At this point a developer stops thinking about one process and starts thinking about tensor, pipeline, and data parallelism.

  • Touchpoint: torchrun, Megatron-LM, DeepSpeed, FSDP config
  • Produces: checkpoints that can later be exported to an inference engine
shell 
torchrun \
  --nproc_per_node=8 \
  --nnodes=2 \
  --node_rank=$NODE_RANK \
  pretrain_gpt.py \
  --tensor-model-parallel-size 8 \
  --pipeline-model-parallel-size 2 \
  --micro-batch-size 2 \
  --global-batch-size 128 \
  --bf16 \
  --use-flash-attn \
  --distributed-backend nccl \
  --save /checkpoints/tiny-gpt
Integration: the training launcher consumes the model from Step 4 and the NCCL fabric from Step 2, then emits checkpoints for inference.
6

Inference — turn checkpoints into an optimized engine

After training, the artifact changes shape: a research checkpoint becomes a serving artifact optimized for latency, throughput, and memory efficiency.

  • Touchpoint: TensorRT-LLM build/export commands or an engine like vLLM
  • Produces: a runnable model package for a serving framework
shell 
python convert_checkpoint.py \
  --input_dir /checkpoints/tiny-gpt \
  --output_dir /artifacts/tiny-gpt-hf

trtllm-build \
  --checkpoint_dir /artifacts/tiny-gpt-hf \
  --output_dir /engines/tiny-gpt-fp8 \
  --gemm_plugin fp8 \
  --max_batch_size 32 \
  --max_input_len 4096 \
  --max_output_len 1024
Integration: this layer consumes training checkpoints and packages them so Triton, NIM, or another serving tier can load them.
7

Orchestration — deploy the serving stack on a cluster

Finally, developers hand the optimized engine to infrastructure tooling so the model becomes a stable service with health checks and autoscaling.

  • Touchpoint: Triton model repository, Kubernetes, DCGM exporter, service endpoints
  • Produces: a production API that callers can hit over HTTP/gRPC
yaml + shell 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: triton-llm
spec:
  template:
    spec:
      containers:
        - name: triton
          image: nvcr.io/nvidia/tritonserver:24.10-py3
          args: ["tritonserver", "--model-repository=/models"]
          resources:
            limits:
              nvidia.com/gpu: 1

curl http://triton-llm.default.svc.cluster.local:8000/v2/models/tiny-gpt/generate \
  -d '{"text_input":"Explain NVLink in one sentence","max_tokens":64}'
Integration: orchestration binds together the inference artifact from Step 6, the GPU nodes from Step 1, and observability such as DCGM and Prometheus.

End-to-end mental model

A developer writes PyTorch code, which dispatches into CUDA / cuBLAS / cuDNN on top of the detected GPU hardware. When that model scales beyond one node, NCCL uses NVLink and InfiniBand to move tensors. A distributed trainer such as Megatron-LM saves checkpoints, TensorRT-LLM converts them into an optimized engine, and Triton + Kubernetes turns that engine into a production endpoint.

Grace CPU & coherent memory (GH200 / GB200)

The Grace CPU is a 72-core Arm Neoverse V2 chip that NVIDIA pairs with Hopper (GH200) or Blackwell (GB200) GPUs over a NVLink-C2C link running at 900 GB/s — roughly 7× faster than PCIe 5.0 x16. Critically, CPU and GPU share a coherent address space, so a kernel can read system memory without explicit copies and CPU offload no longer pays the PCIe tax.

🔗

GH200 Grace-Hopper

1× Grace (72 cores, up to 480 GB LPDDR5X) + 1× H100 (96 GB HBM3). Ideal for workloads with giant KV caches or embedding tables that spill past GPU HBM.

🧱

GB200 NVL72

36× Grace + 72× Blackwell in one liquid-cooled rack, all on a single NVLink 5 domain. The entire rack acts like one huge GPU — TP=72 without leaving NVLink.

💡

What it unlocks

ZeRO-Infinity / DeepSpeed offload runs near-HBM speed. MoE expert parameters can live in LPDDR. Giant inference context windows (millions of tokens) stop being memory-bound.

FP8 training with Transformer Engine — why it matters

FP8 has 2× the throughput of BF16 on Hopper and 4× on Blackwell, but naïve FP8 training diverges because activations have a wider dynamic range than the format can hold. Transformer Engine solves this with per-tensor scaling factors tracked via an exponential moving average of the observed maximum absolute value (amax), then applied with a one-step delay so the scale never depends on the current step's un-seen distribution.

Concretely

Wrap nn.Linear with transformer_engine.pytorch.Linear; wrap the whole transformer block with te.TransformerLayer. Inputs still flow in BF16, the weights and matmul run in FP8, and output gradients round-trip back to BF16. The scaling recipe lives in te.fp8_autocast(fp8_recipe=DelayedScaling(...)). NeMo and Megatron-LM set this up automatically — FSDP works too (enable mixed_precision=MixedPrecision(reduce_dtype=torch.bfloat16)).

🔬Distributed Training Stack

Training a 70B+ parameter LLM requires 3D parallelism across hundreds to thousands of GPUs. These frameworks manage that complexity.

3D Parallelism in one paragraph

Data Parallelism (DP) replicates the model and all-reduces gradients. Tensor Parallelism (TP) shards every layer's weights and all-reduces activations inside each layer — keep TP within one NVLink domain. Pipeline Parallelism (PP) partitions layers across GPUs with interleaved 1F1B microbatches — tolerates slower inter-node links. ZeRO-3 / FSDP shards params, grads, and optimizer state across the DP group — simpler than TP, worse compute-comm overlap. A GPT-3 175B run might use TP=8 · PP=16 · DP=2048: TP inside a DGX node, PP across nodes, DP across the cluster.

Which parallelism strategy should I use?

Enter model size, per-GPU memory, and cluster size. The picker uses a memory budget of 16 bytes/param (BF16 weights + BF16 grads + FP32 Adam states) plus ~1.5 GB/B activations with selective recompute at micro-batch 1 / seq 4K, then picks the simplest strategy that fits with 15% headroom. It assumes full-parameter training — LoRA/QLoRA collapses the optimizer-state term and lets much smaller clusters work.

NVIDIA NIM — the managed microservice form factor

A NIM is an OCI container that wraps Triton + TensorRT-LLM + a specific model behind an OpenAI-compatible API. It starts in seconds, auto-selects the right TRT-LLM engine for the detected GPU, and ships with a health probe and Prometheus metrics endpoint out of the box. Pricing is per-GPU-hour via NVIDIA AI Enterprise licensing.

📦

What's in the box

Pre-quantized FP8/INT4 engine, tokenizer, chat template, function-calling schema, gRPC + REST, Kubernetes Helm chart, Grafana dashboard.

🎨

Model catalog

Llama 3.1/3.3, Mistral/Mixtral, Nemotron, DeepSeek-R1, Stable Diffusion, NV-Embed, Riva ASR/TTS, plus enterprise-partner models (Databricks DBRX, Writer Palmyra).

🤷

Build vs buy

Roll your own Triton+TRT-LLM if you need a custom op or have ML engineers to maintain it. Use NIM if you want an SLA, CVE patches, and no engine-build pipeline.

NVIDIA Dynamo — disaggregated inference at scale

Dynamo is NVIDIA's open-source distributed inference framework aimed at multi-tenant serving clusters. Its defining idea: disaggregate prefill from decode. Prefill (prompt processing) is compute-bound — push it to dense H100s. Decode (token generation) is memory-bandwidth-bound — push it to H200s or B200s with huge HBM. A KV-cache router sends the KV blocks from prefill GPUs to decode GPUs over NVLink or IB, so each workload runs on the GPU best suited to it.

Dynamo also adds smart request routing (prefix-aware to exploit KV-cache hits), elastic scaling (add/remove decode replicas without draining), and multi-model serving via Triton underneath. Pick Dynamo over vLLM when you are running dozens of concurrent models or need sub-linear cost scaling past ~32 GPUs.

Fine-tuning pipeline & multi-LoRA serving

Most production customization is PEFT — Parameter-Efficient Fine-Tuning — not full-weight updates. LoRA trains two small low-rank matrices per layer; QLoRA quantizes the frozen base to 4-bit first; DoRA decomposes updates into magnitude + direction for better quality at the same parameter budget. A 70B LoRA fine-tune fits on one H100 80 GB in hours.

Multi-LoRA at serving time (S-LoRA pattern)

The base model loads once into GPU memory. Each request carries an adapter_id; the engine dynamically loads the tiny LoRA weights for that id from CPU/host memory and fuses them into the matmul. A single 8× H100 node can serve hundreds of customer-specific variants of Llama-3 70B without duplicating the 140 GB base. Supported in vLLM, TensorRT-LLM, and TGI.

The observability loop — DCGM → Prometheus → Grafana → autoscaler

Every node in the AI Factory diagram emits telemetry, and that telemetry is what closes the loop back to scheduling. The canonical pipeline on NVIDIA hardware is DCGM-Exporter (a Prometheus exporter shipped by NVIDIA) scraping per-GPU counters, Prometheus storing them, Grafana visualising, and a controller (HPA, KEDA, or Run:ai policy) acting on the signals.

Metric What it tells you Actionable threshold
DCGM_FI_DEV_SM_ACTIVE Fraction of time SMs had at least one warp resident — your GPU "busy %" < 60% sustained → kernel launch or data-stall bound
DCGM_FI_PROF_PIPE_TENSOR_ACTIVE Tensor-core utilisation — the number that actually maps to useful LLM work < 30% on training → you are not compute-bound; check precision, batch, parallelism
DCGM_FI_DEV_MEM_COPY_UTIL / HBM BW HBM read/write bandwidth used > 85% on inference decode → memory-bandwidth bound (expected); > 85% on training → activation recompute or sharding fix needed
DCGM_FI_DEV_NVLINK_BANDWIDTH_TOTAL NVLink tx+rx bytes — catches mis-placed TP groups across PCIe 0 bytes on a "TP=8" job → topology mis-configured
DCGM_FI_DEV_POWER_USAGE Per-GPU Watts Sustained < 50% TDP while "training" → idle waiting on I/O
DCGM_FI_DEV_XID_ERRORS / ECC Kernel-visible hardware errors, uncorrectable ECC events Any non-zero → drain node, rerun job from last checkpoint
App-level tokens_per_second End-to-end useful throughput from the serving engine Feed directly into HPA / KEDA — scale replicas on tok/s and queue depth, not CPU%

Why not CPU%? LLM serving nodes run the CPU at ~10–20% regardless of load because almost all work is on the GPU. CPU-based autoscaling under-provisions aggressively. Scale on the queue depth reported by Triton / vLLM / Dynamo, or on GPU SM-active averaged over a 30–60 s window.

Fault tolerance: XID / ECC alerts from DCGM are the trigger for NeMo / Megatron-LM's async checkpoint-and-resume. On a SuperPOD, losing a node mid-run is routine; the stack is designed around it. Your checkpoint cadence should be set by MTBF × cluster size, not by instinct — on 10 000 GPUs expect a failure every few hours.

📖Glossary

Quick definitions for terms used throughout the site. Anywhere a term appears inside an <abbr> tag, hovering shows the definition below.

NCCL
NVIDIA Collective Communications Library — implements all-reduce, all-gather, broadcast over NVLink / InfiniBand. The fast path every training framework uses.
SHARP
Scalable Hierarchical Aggregation and Reduction Protocol — performs reductions inside InfiniBand switches, halving all-reduce latency on large clusters.
NVLink
NVIDIA's GPU-to-GPU interconnect. Gen 4 on Hopper (900 GB/s), Gen 5 on Blackwell (1.8 TB/s). Used for tensor-parallel groups inside a node or an NVL72 rack.
NVSwitch
Non-blocking crossbar that joins multiple NVLinks. Extends NVLink from point-to-point to fully-connected 8-GPU (HGX) or 72-GPU (NVL72) domains.
HBM
High-Bandwidth Memory — stacked DRAM co-packaged with the GPU die. H100: 80 GB HBM3 @ 3.35 TB/s; H200: 141 GB HBM3e; B200: 192 GB HBM3e.
TMA
Tensor Memory Accelerator — Hopper feature that asynchronously moves tiles between global and shared memory so SMs don't stall on data.
SM
Streaming Multiprocessor — a GPU's core execution block. H100 has 132 SMs; Blackwell B200 has 148 per die × 2 dies.
MIG
Multi-Instance GPU — hardware partitioning of a single GPU into up to 7 isolated instances, each with its own SMs, L2 cache, and HBM slice.
CUDA
NVIDIA's GPU compute platform and C++ dialect. Everything higher up the stack (cuBLAS, cuDNN, Triton kernels, PyTorch ops) ultimately compiles to CUDA PTX.
PTX
Parallel Thread eXecution — NVIDIA's virtual ISA. CUDA compiles to PTX, which the driver JIT-compiles to hardware SASS for a specific GPU generation.
cuBLAS / cuDNN
NVIDIA's hand-tuned libraries for dense linear algebra (cuBLAS) and neural-net primitives (cuDNN). Used under the hood by every major framework.
Transformer Engine
NVIDIA library that adds FP8 auto-scaling to attention & MLP blocks, used by NeMo, Megatron-LM, and PyTorch's te.Linear.
FP8 / E4M3 / E5M2
8-bit floating-point formats. E4M3 (4 exponent, 3 mantissa) is used for weights/activations in forward/backward; E5M2 for gradients. Halves memory and doubles Tensor Core throughput vs BF16.
BF16
Brain Float 16 — 8 exponent, 7 mantissa bits. Same range as FP32, lower precision. The default training dtype on modern GPUs.
KV-cache
Per-layer cache of attention keys & values retained across decode steps. Dominates inference memory once sequence length grows.
Prefill
The first inference pass, where the whole prompt is consumed and the KV-cache is populated. Compute-bound, parallel across tokens.
Decode
Token-by-token generation after prefill. Memory-bandwidth-bound because every step reads the full KV-cache.
TTFT
Time To First Token — end-to-end latency until the first output token. Dominated by prefill cost for short prompts; by queuing for busy servers.
TPOT / ITL
Time Per Output Token (a.k.a. Inter-Token Latency). Dominated by decode memory bandwidth; roughly KV_bytes / HBM_BW per layer.
Continuous batching
Scheduling technique (vLLM, TGI, TRT-LLM) that adds new requests into a running batch each decode step, instead of waiting for a static batch to finish.
PagedAttention
vLLM's KV-cache allocator — stores KV in fixed-size pages, eliminating fragmentation and enabling prefix sharing.
Speculative decoding
Use a small draft model to propose k tokens, verify them in one big-model forward pass. 2–3× decode speedup with no accuracy loss when drafts accept.
MoE
Mixture of Experts — a layer where a router picks top-k experts per token. Grows parameter count without matching FLOP cost. Mixtral, DeepSeek-V3, GPT-4 all use MoE.
DP / TP / PP
Data Parallel / Tensor Parallel / Pipeline Parallel — the three axes of 3D parallelism. DP replicates, TP shards weights across GPUs, PP shards layers across stages.
ZeRO
Zero Redundancy Optimizer — DeepSpeed's staged sharding of optimizer state (Z1), gradients (Z2), and weights (Z3) across DP ranks. Equivalent to FSDP.
FSDP
Fully Sharded Data Parallel — PyTorch-native equivalent of ZeRO-3. Shards weights, grads, and optimizer states; gathers on-demand during forward/backward.
LoRA / QLoRA
Low-Rank Adaptation — freeze the base model and train small A·B rank-r matrices. QLoRA quantizes the frozen base to 4-bit NF4. Makes fine-tuning 70B feasible on a single 80 GB GPU.
DoRA
Weight-Decomposed LoRA — splits weight updates into magnitude and direction; closes the accuracy gap between LoRA and full fine-tune.
S-LoRA
Multi-LoRA serving pattern — hold hundreds of LoRA adapters in GPU memory and route each request to the right one, amortising the base model.
AWQ / GPTQ
Post-training weight quantization to INT4. AWQ preserves salient-channel scale; GPTQ uses layer-wise Hessian-guided rounding. Both land within ~1 pp of FP16 on most LLMs.
SmoothQuant
INT8 PTQ that migrates activation outliers into weights via per-channel scaling — keeps INT8 accuracy on models with heavy activation tails.
FlashAttention
Tiled, IO-aware attention kernel (Tri Dao et al.) — fuses softmax and the matmuls so that activations never materialize in HBM. 2–4× speedup, exact.
DCGM
Data Center GPU Manager — NVIDIA's telemetry & diagnostics daemon. The dcgm-exporter ships per-GPU counters to Prometheus.
GPUDirect RDMA / Storage
Lets the NIC or NVMe device DMA straight into GPU HBM, skipping host memory. Essential for IB-attached training and checkpoint I/O.
Magnum IO
Umbrella brand covering NVIDIA's I/O libraries: NCCL, GPUDirect, NVSHMEM, UCX. Think "the data-movement layer."
NVL72
A single rack of 72 Blackwell GPUs connected by 5th-gen NVLink into one coherent domain — 72-way tensor parallel without ever touching the network.
BlueField DPU
Data Processing Unit — SmartNIC with ARM cores. Offloads networking, storage, and security from the host so the x86/Grace CPU can focus on ML workloads.
Spectrum-X
NVIDIA's Ethernet-based AI fabric — Spectrum-4 switch + BlueField-3 DPU + adaptive routing. RoCE-grade performance on standard Ethernet operations.
InfiniBand NDR
400 Gb/s per-port lossless fabric used in DGX SuperPOD. Pairs with SHARP for in-network reductions.
NIM
NVIDIA Inference Microservice — OCI container bundling Triton + TRT-LLM + a model + OpenAI-compatible API. The enterprise "drop-in" serving unit.
Dynamo
NVIDIA's disaggregated inference framework — separates prefill and decode workers, routes KV-cache between them. Targets multi-tenant scale.
NGC
NVIDIA GPU Cloud — NVIDIA's registry of optimised containers and model checkpoints. Monthly releases lock driver + CUDA + framework combos.
